# شرح مفصّل — cloudflare-worker/

هذا المجلد هو **الخلفية الكاملة (backend)** للموقع. لا يوجد أي سيرفر آخر — كل منطق الكتابة/القراءة الديناميكي يمر من هنا. يُنشر بشكل منفصل تماماً عن Replit، عبر `wrangler` أو مباشرة من لوحة تحكم Cloudflare (Workers & Pages → Edit Code). راجع `PROJECT.md` في جذر المشروع لخطوات النشر التفصيلية.

## الملفات

| الملف | الغرض |
|---|---|
| `upload-worker.js` | كل كود الـ Worker: نقاط الـ API، رفع الصور، مزامنة R2 |
| `schema.sql` | تعريف جداول قاعدة بيانات D1 (`driverjo-db`) |
| `wrangler.toml` | إعدادات النشر: اسم الـ Worker، ربط D1، ربط R2 |

## البنية التقنية (`wrangler.toml`)

- اسم الـ Worker: `upload` → يعمل على `https://upload.idriverjo.workers.dev`
- **R2 bucket**: `driverjo-data` (binding: `BUCKET`) — يخزّن ملفات JSON العامة وصور الأسئلة/المراكز، ويُقرأ علنياً عبر النطاق المخصص `https://data.driverjo.online` (متغيّر `PUBLIC_BASE_URL`).
- **D1 database**: `driverjo-db` (binding: `DB`) — قاعدة البيانات الحقيقية لكل شيء (مراكز، طلبات انضمام، محافظات، مناطق، أسئلة).

## قاعدة البيانات D1 (`schema.sql`)

كل الجداول تتبع نفس النمط: عمود `id` كمفتاح أساسي، عمود `data` نصّي يحتوي **كامل السجل كـ JSON** (يحاكي شكل Firebase RTDB القديم `{id: {...fields}}`)، بالإضافة لأعمدة إضافية للفهرسة فقط:

- **`governorates`**: `id`, `data` (`{ name, archived? }`), `updated_at`
- **`areas`**: `id`, `governorate_id` (مفهرس)، `data` (`{ governorateId, name, archived? }`), `updated_at`
- **`centers`**: `id`, `data` (السجل الكامل للمركز: الاسم، العنوان، `lat`/`lng`، جدول العمل، هل هو مروَّج `promoted`...)، `updated_at`
- **`center_requests`**: `id`, `status` (مفهرس، القيمة الافتراضية `pending`)، `data` (نفس شكل سجل المركز تقريباً، بالإضافة لبيانات مقدّم الطلب)، `updated_at`
- **`questions`**: `id`, `category` (مفهرس)، `data` (`{ category, question, options, correctAnswer, mediaType, mediaUrl? }`), `updated_at`

**لماذا هذا التصميم؟** لتقليل عدد التعديلات المطلوبة عند إضافة حقل جديد — تضيف الحقل داخل الـ JSON مباشرة (في الواجهة الأمامية) دون الحاجة لتعديل بنية الجدول (schema migration)، إلا إذا احتجت فهرسة/تصفية بهذا الحقل تحديداً (عندها فقط تضيف عموداً منفصلاً + `CREATE INDEX`).

## المصادقة (Authentication)

نوعان من المفاتيح، يُرسلان كترويسات HTTP (headers) من الواجهة الأمامية:

1. **`X-Upload-Key`**: يُتحقق منه مباشرة عند رفع صورة (الطلب على جذر الـ Worker `POST /`). يُقارَن بـ `env.UPLOAD_KEY`.
2. **`X-Admin-Key`**: يُتحقق منه عبر الدالة المساعدة `isAdmin(request, env)` لأي عملية كتابة/حذف/تعديل تحت `/api/*` (الإضافة، التعديل، الحذف، الموافقة على طلب، وإعادة المزامنة اليدوية). كل عمليات **القراءة (GET)** تحت `/api/*` عامة ولا تتطلب أي مفتاح.

كِلا المفتاحين مكتوبان مباشرة (hardcoded) داخل الصفحات التي تحتاجهما في الواجهة الأمامية (`admin.html` لمفتاح الأدمن، و`admin.html`/`center-join.html` لمفتاح الرفع) — هذا مقبول كتنازل مقصود بما أن الموقع بلا سيرفر خلفي خاص به يمكنه إخفاء هذه المفاتيح، ولأن الأثر الأسوأ لتسريبها محدود (كتابة بيانات وليس اختراق حساب).

## نقاط الـ API (Endpoints)

جميعها تحت `https://upload.idriverjo.workers.dev`. الأقواس المربعة `[]` تعني اختياري.

### رفع الصور — `POST /`
- المصادقة: `X-Upload-Key`
- المدخل: إمّا `multipart/form-data` (ملف صورة) أو JSON بصيغة `{ "url": "..." }`
- يخزّن الصورة في R2 (تحت مسار مثل `questions/...`) ويُرجع `{ url }` برابط عام دائم.

### المحافظات — `/api/governorates`
- `GET` (عام): جلب كل المحافظات
- `POST` / `PATCH /:id` / `DELETE /:id` (أدمن): إضافة/تعديل/حذف
- بعد أي كتابة: يُعاد توليد `governorates.json` في R2 تلقائياً (`syncGovernorates`)

### المناطق — `/api/areas`
- `GET` (عام، يقبل فلترة `?governorateId=...`)
- `POST` / `PATCH /:id` / `DELETE /:id` (أدمن)
- بعد أي كتابة: تحديث `areas.json` تلقائياً (`syncAreas`)

### المراكز — `/api/centers`
- `GET` (عام، بالكامل أو `GET /:id` لمركز واحد)
- `POST` / `PATCH /:id` / `DELETE /:id` (أدمن)
- بعد أي كتابة: تحديث `centers.json` تلقائياً (`syncCenters`) — **هذا هو الملف الذي تقرأ منه `driving-schools.html` مباشرة**

### طلبات الانضمام — `/api/center-requests`
- `POST` (عام، بلا مصادقة — أي زائر يمكنه إرسال طلب انضمام جديد)
- `GET` / `PATCH /:id` / `DELETE /:id` (أدمن، لمراجعة/تعديل/حذف الطلبات)
- `POST /:id/approve` (أدمن): عملية خاصة تُنشئ مركزاً جديداً في جدول `centers` من بيانات الطلب **وتُحدّث حالة الطلب إلى `approved`** في خطوة واحدة ذرّية (atomic) — هذا مسار الموافقة الفعلي الذي تستخدمه لوحة التحكم.

### الأسئلة — `/api/questions`
- `GET` (عام)
- `POST` / `PATCH /:id` / `DELETE /:id` (أدمن)
- بعد أي كتابة: تحديث `questions.json` تلقائياً (`syncQuestions`)

### حل رابط خرائط جوجل — `POST /api/resolve-maps-link`
- عام (بلا مصادقة)
- المدخل: `{ "url": "<رابط Google Maps، مختصر أو كامل>" }`
- **لماذا وُجدت**: روابط Google Maps المختصرة (`maps.app.goo.gl/...`) لا تحمل الإحداثيات داخل نص الرابط نفسه؛ الإحداثيات تظهر فقط بعد إعادة توجيه (HTTP redirect) لا يستطيع المتصفح قراءتها بسبب قيود CORS على الروابط المختصرة. هذه النقطة تحل المشكلة من طرف السيرفر (لا قيود CORS هناك).
- **كيف تعمل**: تتبع سلسلة إعادة التوجيه (مهلة أقصاها 10 ثوانٍ)، ثم تبحث عن الإحداثيات في الرابط النهائي بعد إعادة التوجيه **وأيضاً** داخل محتوى الصفحة نفسها (body)، باستخدام دالة `extractLatLng()` التي تجرّب عدة أنماط بالترتيب:
  1. `@lat,lng` (نمط رابط الخريطة القياسي)
  2. `?q=lat,lng`
  3. `?ll=lat,lng`
  4. `!3dlat!4dlng` (نمط بيانات المكان الداخلي في روابط جوجل)
- المخرج: `{ lat, lng, resolvedUrl }`

**من يستخدم هذه النقطة؟** `admin.html` (عند إضافة/تعديل مركز، وأيضاً زر "تحديث الإحداثيات" الذي يفحص كل المراكز الناقصة الإحداثيات ويُصلحها دفعة واحدة) و`center-join.html` (عند إرسال طلب انضمام جديد) — كلاهما يستدعي هذه النقطة **قبل** حفظ/إرسال البيانات، بحيث تُحفظ الإحداثيات (`lat`/`lng`) جاهزة في قاعدة البيانات ولا تحتاج أي حساب لاحق عند العرض.

### إعادة المزامنة اليدوية — `POST /api/resync`
- أدمن فقط
- يُعيد توليد **كل** ملفات JSON في R2 (`centers.json`, `governorates.json`, `areas.json`, `questions.json`) من محتوى D1 الحالي دفعة واحدة.
- **متى تُستخدم**: بعد أي تعديل يدوي مباشر على D1 يتجاوز الـ API (مثلاً عبر `wrangler d1 execute`)، لأن مثل هذا التعديل لا يُشغّل دوال المزامنة التلقائية.

## آلية المزامنة مع R2 (كيف تصل البيانات للصفحات العامة)

1. أي عملية كتابة ناجحة على `/api/governorates`, `/api/areas`, `/api/centers`, أو `/api/questions` تستدعي تلقائياً `syncSnapshot(env, table, filename)`.
2. `syncSnapshot` تقرأ **كل صفوف الجدول** من D1، تحوّلها لقاموس `{ id: {...data} }` (مطابق لشكل Firebase القديم تماماً حفاظاً على التوافق مع الصفحات العامة)، وترفعه كملف JSON إلى R2.
3. بعد كل مزامنة، تُستدعى `bumpVersion(env)` التي تكتب `{ v: Date.now() }` إلى `version.json` في R2 — هذا يُستخدم من قبل الصفحات العامة (وقالب Blogger) كآلية "كسر التخزين المؤقت" (cache-busting) للتأكد من جلب أحدث نسخة من ملفات JSON.
4. الصفحات العامة تقرأ فقط من `https://data.driverjo.online/<filename>.json` (النطاق المخصص لـ R2)، وليس من D1 أو الـ Worker مباشرة.

**تنبيه على التأخير**: نطاق `data.driverjo.online` عبر R2 لديه ذاكرة تخزين مؤقت على حافة الشبكة (edge cache) لمدة ~60 ثانية تقريباً، لذلك أي تعديل حديث عبر لوحة التحكم قد لا يظهر فوراً للزوار.

## CORS

مسموح فقط لأصل الطلبات القادمة من:
- `https://www.driverjo.online`
- `https://driverjo.online`

الطرق المسموحة: `GET, POST, PATCH, DELETE, OPTIONS`. الترويسات المسموحة: `Content-Type, X-Upload-Key, X-Admin-Key`. **ملاحظة**: هذا يعني أن اختبار نقاط الكتابة من بيئة معاينة محلية (مثل معاينة Replit) سيفشل بخطأ CORS — هذا متوقع وليس عطلاً، لأن نطاق المعاينة ليس ضمن القائمة البيضاء.
