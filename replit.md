# driverjo — نظرة عامة على المشروع

موقع مدرسة قيادة/اختبار نظري (Blogger theme) مستضاف خارجيًا على driverjo.online، لا يوجد سيرفر Replit ولا workflow — القالب (`template`) وصفحات `pages/*.html` تُرفع يدويًا لـ Blogger. البيانات (الأسئلة، المراكز، المحافظات، المناطق، المستخدمون، التقييمات، طلبات الانتساب) في Firebase Realtime Database، ويُعاد نشرها كـ JSON ثابت على Cloudflare (`data.driverjo.online`) لتسريع القراءة العامة.

## أمان قاعدة بيانات Firebase
- تسجيل دخول لوحة الإدارة (`pages/admin.html`) أصبح مصادقة حقيقية (Firebase Auth بالبريد/كلمة المرور) بدل إدخال UID يدوي. عقدة `admin/{uid}` لا تزال تُستخدم كقائمة سماح، لكن الـ `{uid}` يجب أن يطابق UID حقيقي من Firebase Authentication.
- قواعد قاعدة البيانات الموصى بها موجودة في `firebase-database.rules.json` (تُطبَّق يدويًا من Firebase Console > Realtime Database > Rules — لا يوجد نشر تلقائي من هذا المستودع). ملخص السياسة: قراءة عامة لـ questions/centers/governorates/areas/reviews، كتابة مقصورة على مشرفين مصرَّح لهم لهذه العقد، مع استثناء إنشاء عناصر جديدة (بدون تعديل/حذف) بدون تسجيل دخول لـ CenterRequests وareas (يطابق سلوك الموقع الحالي في `center-join.html`)، وusers/{uid} يقرأ/يكتب فقط من صاحبه أو المشرف.
- قبل تفعيل القواعد الجديدة: يجب إنشاء حسابات Firebase Authentication حقيقية (بريد/كلمة مرور) للمشرفين من Firebase Console، ثم تحديث عقدة `admin/{uid}` بالـ UID الحقيقي الجديد (القيم القديمة كانت أي نص عشوائي وليست UID حقيقي فعلي).

## أداة ترحيل الصور القديمة
- زر "ترحيل الصور القديمة" في صفحة إدارة الأسئلة (`admin.html`) يفحص كل الأسئلة، ويحوّل أي `mediaUrl` بصيغة base64 أو رابط خارجي (غير مستضاف على `data.driverjo.online`) إلى ملف على Cloudflare R2، ثم يحدّث السؤال في Firebase تلقائيًا.
- الروابط الخارجية تُرحَّل عبر وضع جديد في الـ Worker (JSON `{url}`) يقوم بتحميل الصورة من جهة السيرفر لتفادي قيود CORS في المتصفح.
- الترحيل يعمل عنصرًا بعنصر مع مهلة زمنية (timeout) لكل طلب لتفادي التعليق، وسجل مرئي لكل نجاح/فشل. لا يوجد تراجع (undo) تلقائي — المستخدم اختار عدم إضافة تصدير نسخة احتياطية مسبقة.
- محدودية معروفة: إن فشل تحديث Firebase بعد نجاح الرفع لـ R2، يبقى الملف المرفوع على R2 دون استخدام (لا حذف تلقائي لليتيم).

## رفع الصور عبر Cloudflare
- `pages/admin.html` يرفع صور/GIF الأسئلة إلى Cloudflare Worker (`cloudflare-worker/upload-worker.js`) المرتبط بـ R2 bucket، ويخزّن الرابط العام الناتج كـ `mediaUrl` في Firebase بدل base64.
- تم اختيار هذا الحل بعد التأكد (باختبار فعلي محذوف لاحقًا) أن Blogger API v3 لا يمكنه استضافة/إعادة استضافة الصور لتطبيقات خارجية — يُخزّن base64 خامًا كما هو.
- ملاحظة أمنية مقبولة: `UPLOAD_KEY` مفتاح ثابت مُضمَّن في كود العميل (وليس سرًا حقيقيًا)، يعتمد الحد من إساءة الاستخدام على تقييد CORS في الـ Worker لنطاقات driverjo فقط.

## تفضيلات المستخدم
- لا تغييرات في هيكلة المشروع الحالية (يبقى Blogger + Firebase) إلا بطلب صريح.
