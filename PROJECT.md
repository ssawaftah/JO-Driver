# دليل مشروع driverjo — نظرة شاملة

هذا الملف هو المرجع الرئيسي لفهم بنية المشروع بالكامل. إذا كنت تريد تفاصيل أعمق عن جزء معيّن:
- تفاصيل الـ Cloudflare Worker وقاعدة البيانات: `cloudflare-worker/README.md`
- تفاصيل كل صفحة HTML على حدة: `pages/README.md`

---

## 1. ما هو هذا المشروع؟

موقع دليل مدارس تعليم القيادة في الأردن، باللغة العربية، اسمه **driverjo**. يحتوي على:
- صفحة تصفح وبحث عن مراكز تعليم القيادة حسب المحافظة والمنطقة.
- صفحة تفاصيل لكل مركز (ساعات العمل، الصور، رقم الهاتف/واتساب، الموقع على الخريطة).
- نموذج عام يسمح لأصحاب مراكز تعليم القيادة بالتقديم للانضمام إلى الموقع.
- لوحة تحكم إدارية (Admin Panel) لمراجعة طلبات الانضمام والموافقة عليها، وإدارة المراكز والمحافظات والمناطق والأسئلة.
- منصّة أسئلة اختبار نظري (تدريب + اختبار) لرخصة القيادة الأردنية.
- صفحة تقييمات (reviews) للمستخدمين.

## 2. أين يُستضاف الموقع فعلياً؟ (مهم جداً)

**الموقع الحقيقي يعمل على Blogger، وليس على Replit.**

- هذا الـ Repl هو بيئة **تحرير ومعاينة فقط**. الـ workflow المُشغَّل هنا (`npx serve pages -p 5000`) هو مجرد سيرفر ملفات ثابتة (static file server) يعرض محتوى مجلد `pages/` لمعاينته أثناء التطوير، ولا يعكس طريقة الاستضافة الحقيقية.
- كل ملف داخل `pages/*.html` يُنسخ محتواه بالكامل ويُلصق يدوياً داخل صفحة/تدوينة مطابقة في Blogger (تحرير HTML مباشرة).
- ملف `template` (بحجم ~800KB) هو **قالب Blogger XML** (ثيم مدفوع من fineshopdesign.com اسمه "plus-ui"، إصدار 3.7.0) وهو الهيكل العام للموقع (الهيدر، الفوتر، التصميم، القوائم...). هذا الملف **منفصل تماماً** عن صفحات `pages/*.html` من الناحية التقنية — لا تتضمّنه صفحات `pages/` ولا يتضمّنها، بل صفحات `pages/` تُلصق كمحتوى داخل صفحات/تدوينات منفصلة تستخدم هذا القالب كإطار عام للموقع. **لم يتم تعديل هذا الملف في أي عمل سابق على هذا المشروع.**
- **النتيجة العملية**: لأن الموقع بلا سيرفر خلفي (no backend) في الإنتاج، كل صفحات `pages/*.html` مبنية بـ HTML/CSS/JavaScript خام بدون أي خطوة بناء (build step)، وكل منطق التطبيق (fetch، حسابات، عرض) يعمل بالكامل داخل المتصفح (client-side).
- أي ميزة جديدة يجب أن تُصمَّم بافتراض عدم وجود سيرفر خاص بنا يشغّل كوداً — المصدر الوحيد للبيانات الديناميكية هو Cloudflare (Worker + D1 + R2) و/أو Firebase (لأجزاء محدودة، انظر القسم 4).

## 3. خريطة المجلدات

```
/
├── pages/                    → كل صفحات الموقع (HTML+CSS+JS)، تُنسخ يدوياً إلى Blogger
│   ├── admin.html            → لوحة التحكم الإدارية
│   ├── center-join.html      → نموذج "انضم كمركز" العام
│   ├── driving-schools.html  → صفحة قائمة/بحث المراكز
│   ├── center-details.html   → صفحة تفاصيل مركز واحد
│   ├── theory-test-practice.html      → منصة التدريب/الاختبار النظري
│   ├── driving-theory-questions.html  → بنك أسئلة النظري (تصفح/فئات)
│   ├── reviews.html          → صفحة التقييمات
│   └── road-safety.html      → محتوى فئة "السلامة المرورية" ضمن النظري
├── cloudflare-worker/        → الخلفية الوحيدة للموقع (Worker + D1 + R2)، تُنشر عبر wrangler بشكل منفصل
│   ├── upload-worker.js      → كل كود الـ Worker (API + رفع صور + مزامنة R2)
│   ├── schema.sql            → تعريف جداول قاعدة بيانات D1
│   ├── wrangler.toml         → إعدادات النشر (اسم الـ Worker، ربط D1/R2)
│   ├── seed.sql / seed-questions.sql → نسخ ترحيل تاريخية من Firebase إلى D1 (يمكن حذفها لاحقاً)
├── template                  → قالب Blogger XML (ثيم الموقع العام، منفصل عن pages/)
├── replit.md                 → ملخص مختصر للمشروع وتفضيلات المستخدم (يُقرأ تلقائياً من قبل الوكيل)
└── .replit                   → إعداد الـ workflow الخاص بالمعاينة داخل Replit فقط
```

## 4. من أين تأتي البيانات؟ (مصدرا البيانات في المشروع)

المشروع يستخدم **مصدرين منفصلين للبيانات**، ولكل منهما دور محدد ولا يجب الخلط بينهما:

### أ) Cloudflare (المصدر الرئيسي — المراكز والمحافظات والمناطق وطلبات الانضمام والأسئلة)
- **قاعدة البيانات الفعلية**: Cloudflare D1، اسمها `driverjo-db`.
- **كل القراءة/الكتابة** تمر حصراً عبر الـ Worker المنشور على `https://upload.idriverjo.workers.dev` (نقاط API مفصّلة في `cloudflare-worker/README.md`).
- بعد كل عملية كتابة (إضافة/تعديل/حذف)، الـ Worker يُعيد توليد نسخة JSON مطابقة للجدول ويرفعها إلى **R2** (bucket اسمه `driverjo-data`)، والتي تظهر للعامة على `https://data.driverjo.online/<اسم-الملف>.json` (مثال: `centers.json`, `governorates.json`, `areas.json`, `questions.json`).
- صفحات القراءة العامة (`driving-schools.html`, `center-details.html`, صفحات النظري) **لا تتصل بالـ Worker مباشرة** — هي تقرأ فقط من ملفات JSON الثابتة على `data.driverjo.online` (أسرع وأرخص، ولأن Blogger لا يحتاج مصادقة لهذه القراءات).
- صفحة `admin.html` تتصل بالـ Worker مباشرة لأنها تحتاج كتابة (تتطلب `X-Admin-Key`)، وكذلك `center-join.html` عند إرسال طلب انضمام جديد (`POST /api/center-requests`، بلا مصادقة لأنه نموذج عام).
- **ملاحظة تخزين مؤقت (Cache)**: نطاق `data.driverjo.online` (عبر R2) لديه ذاكرة تخزين مؤقت على حافة الشبكة (edge cache) مدتها ~60 ثانية، لذلك أي تعديل عبر لوحة التحكم قد يستغرق حتى دقيقة ليظهر للزوار العاديين.

### ب) Firebase (محدود — لم يعد المصدر الرئيسي)
بعد ترحيل بيانات المراكز إلى Cloudflare، لم يتبقَّ لـ Firebase سوى دورين محدودين:
1. **تسجيل دخول لوحة التحكم** (`admin.html`): تسجيل الدخول (بريد/كلمة مرور) والتحقق من صلاحية "admin" يتم عبر Firebase Auth + Firebase Realtime Database (جدول `users`)، عبر كائن مساعد اسمه `JDAuth`.
2. **صفحة التقييمات** (`reviews.html`): بيانات التقييمات نفسها وتسجيل الدخول بها (Google Auth) ما زالا في Firebase بالكامل، لم يُنقلا إلى D1.

## 5. الخرائط (Google Maps)

- كلٌّ من `center-join.html` (النموذج العام) و`admin.html` (نموذج إضافة/تعديل مركز في اللوحة) يستخدمان **Google Maps JavaScript API** (Maps + Places + Geocoding) لاختيار موقع المركز، بديلاً عن تطبيق سابق كان يستخدم Leaflet/OpenStreetMap.
- الطريقة الفعلية المستخدمة هي "الرابط أولاً" (link-first): المستخدم يلصق رابط Google Maps للمركز، والصفحة تحاول استخراج الاسم والإحداثيات من الرابط مباشرة بدون استدعاء Places API.
- **مشكلة تم اكتشافها وإصلاحها**: روابط Google Maps المختصرة (مثل `maps.app.goo.gl/...`) لا تحمل الإحداثيات داخل الرابط نفسه — تظهر الإحداثيات فقط بعد إعادة التوجيه (redirect) التي لا يستطيع المتصفح قراءتها بسبب قيود CORS. الحل: نقطة API جديدة في الـ Worker اسمها `POST /api/resolve-maps-link` تتبع إعادة التوجيه من طرف السيرفر (لا حدود CORS هناك) وتُرجع الإحداثيات الحقيقية. تفاصيل كاملة في `cloudflare-worker/README.md`.
- لأن الموقع بلا سيرفر خلفي خاص، مفتاح Google Maps API مكتوب مباشرة (hardcoded) داخل `center-join.html` و`admin.html` — وهذا أمر معتاد ومقبول لمفاتيح خرائط جوجل التي تُستخدم من المتصفح (يُنصح بتقييد المفتاح بالنطاق عبر Google Cloud Console).

## 6. رفع الصور

نمط موحّد يُستخدم في أكثر من مكان (`admin.html`, `center-join.html`):
1. المستخدم يسحب/يختار صورة.
2. الصورة تُصغَّر/تُعاد ضغطها في المتصفح قبل الرفع (لتقليل الحجم).
3. تُرفع عبر `POST` إلى جذر الـ Worker (`upload.idriverjo.workers.dev/`) مع ترويسة `X-Upload-Key`.
4. الـ Worker يخزّنها في R2 ويُرجع رابطاً عاماً للصورة يُحفظ ضمن بيانات المركز/السؤال.

## 7. النشر والتحديث — ملخص عملي

| الجزء | كيف يُحدَّث | أين |
|---|---|---|
| صفحات الموقع (`pages/*.html`) | نسخ محتوى الملف كاملاً ولصقه في تحرير HTML للصفحة المطابقة | Blogger |
| قالب الموقع العام (`template`) | نادراً ما يتغيّر؛ إذا تغيّر، يُستورد كـ XML كامل من إعدادات الثيم | Blogger (Theme) |
| الـ Worker (`upload-worker.js`) و D1/R2 | `wrangler deploy` من داخل مجلد `cloudflare-worker/` (عبر Terminal محلي، أو مباشرة من Cloudflare Dashboard → Workers & Pages → Edit Code) | Cloudflare |
| قاعدة بيانات D1 (تغييرات على الجداول نفسها، schema) | `wrangler d1 execute driverjo-db --file=schema.sql` (أو أمر SQL مباشر) | Cloudflare |

**تذكير مهم**: أي تعديل على `upload-worker.js` يجب نشره عبر Cloudflare أولاً، ثم تحديث أي صفحة HTML تعتمد على السلوك الجديد في Blogger — وليس العكس، لتفادي استدعاء نقاط API غير موجودة بعد على الخادم الحقيقي.

## 8. تفضيلات ثابتة يجب مراعاتها عند التطوير مستقبلاً

- لا تفترض وجود سيرفر خلفي مستضاف على Replit في بيئة الإنتاج — أي منطق سيرفر يجب أن يكون داخل `cloudflare-worker/upload-worker.js`.
- لا تُعدّل ملف `template` إلا إذا طُلب ذلك صراحةً — هو قالب Blogger مدفوع ومنفصل عن صفحات `pages/`.
- عند إضافة حقل بيانات جديد لمركز/سؤال/محافظة، يجب تحديث: (1) `schema.sql` إن لزم عمود مفهرس جديد، (2) منطق `admin.html` (النموذج + العرض)، (3) أي صفحة عامة تعرض هذا الحقل، (4) التأكد من أن `syncSnapshot` في الـ Worker يعكس الحقل تلقائياً (لأنه يخزّن الكائن كاملاً كـ JSON، عادة لا حاجة لتغييره).
- ملفات `cloudflare-worker/seed.sql` و`seed-questions.sql` هي نسخ ترحيل تاريخية (Firebase → D1) للأرشفة فقط، يمكن حذفها بأمان لاحقاً.
